# R-05 — Particle system viability

## TL;DR
Do **not** adopt `three-nebula` for PANTHÉON v1: last npm publish was November 2021, last repo commit August 2022, it bundles its own copy of three.js (bundle-bloat risk), has no official TypeScript types or R3F wrapper, and its GPURenderer is documented as broken on iOS Safari — the exact mobile floor we must hit. Use a **hand-rolled layer built on `THREE.InstancedMesh` + `THREE.Points`** for Anansi/Brigid/Susanoo kits and generic combat VFX; it is the controllable, mobile-safe path that matches our per-god 8 MB asset budget and 30 FPS iPhone 12 / Pixel 6 floor (DESIGN_DOCUMENT.md §1, §7). Keep `three.quarks` as a watch-list fallback if Phase 2 authoring velocity becomes a blocker — it is actively maintained, TypeScript-first, and ships an R3F integration.

## three-nebula status (as of 2026-04-24)

- **npm version & publish date**: `three-nebula@10.0.3`, published 2021-11-05; no release in > 4 years — [npm registry metadata](https://registry.npmjs.org/three-nebula) (accessed 2026-04-24).
- **Repo activity**: GitHub master's last commit is 2022-08-07 ("Particle rotation and graceful emitter EOL"); the four preceding commits span Nov 2021 → Jan 2021 — [commits view](https://github.com/creativelifeform/three-nebula/commits/master) (accessed 2026-04-24).
- **Issue velocity**: 34 open issues (including #260 filed 2026-03-29) against a long tail of closed issues dating back to 2020–2022 and a "Future of the library" meta-issue (#220, opened 2022-07-22) asking whether the project is abandoned — [issues list](https://github.com/creativelifeform/three-nebula/issues), [issue #220](https://github.com/creativelifeform/three-nebula/issues/220) (accessed 2026-04-24).
- **Maintenance signal**: Snyk rates it 48/100, flags "No commits over the last 6 months" and "no new versions released in the past 12 months", weekly downloads ~768 — [Snyk advisor](https://security.snyk.io/package/npm/three-nebula) (accessed 2026-04-24).
- **TypeScript**: No shipped `.d.ts`. `@types/three-nebula@10.0.5` exists on DefinitelyTyped (published 2025-01-02, ~107 KB unpacked, maintained by a single community author `moonraker22`) — [@types/three-nebula registry](https://registry.npmjs.org/@types/three-nebula) (accessed 2026-04-24); confirmed third-party by [moonraker22/three-nebula-types](https://github.com/moonraker22/three-nebula-types) (accessed 2026-04-24).
- **Bundle**: Tarball unpacked size 2.91 MB, 238 files — [registry v10.0.3](https://registry.npmjs.org/three-nebula/10.0.3) (accessed 2026-04-24). Historically the package **shipped a copy of three.js inside its bundle**, causing webpack to include three.js twice (~800 KB → 1.4 MB regression) — [issue #54](https://github.com/creativelifeform/three-nebula/issues/54) (accessed 2026-04-24). Even if fixed in current tarball, our build must prove it via bundle analyzer before merging.
- **Architecture**: Primarily **CPU-side per-particle update loop** that writes sprite positions/rotations/colors each frame; three renderer classes (`SpriteRenderer`, `MeshRenderer`, `GPURenderer`). Performance-improvement issue #65 documents that the CPU update path iterates the full particle array per frame and burns cycles on no-op sprite rotation — [issue #65](https://github.com/creativelifeform/three-nebula/issues/65) (accessed 2026-04-24). It does **not** use `THREE.InstancedMesh` under the hood; sprites/meshes are added as individual Object3D children, so 1000 particles imply a draw call per particle unless batched manually via `MeshRenderer` targets.
- **Mobile Safari**: `GPURenderer` is documented **incompatible with iOS Safari** because it relies on floating-point textures that iOS/WebKit doesn't support — [issue #127](https://github.com/creativelifeform/three-nebula/issues/127) (accessed 2026-04-24). Fallback is `SpriteRenderer`, which pushes the load back onto the CPU — the exact opposite of what a mobile floor wants. General Safari WebGL pain (UBO emulation, 150 ms upload hitches) is confirmed by Wonderland Engine's writeup — [WebGL Performance on Safari](https://wonderlandengine.com/news/webgl-performance-safari-apple-vision-pro/) (accessed 2026-04-24).
- **R3F integration**: No official wrapper. The canonical pattern in the pmndrs discussion is a singleton `NebulaEngine` class, manual `Nebula.fromJSONAsync()` load, and a `useFrame` tick calling `system.update()` — [react-three-fiber discussion #1658](https://github.com/pmndrs/react-three-fiber/discussions/1658) (accessed 2026-04-24). The original question is still marked "Unanswered".

## Hand-rolled InstancedMesh / Points trade-off

Our v1 VFX list (from DESIGN_DOCUMENT.md §6 per-god kits and §8 combat feel stack):

- **Anansi**: web-strand projectile (stretched quad / thin tube), dome shimmer (translucent instanced shards + scroll uv), silk-trap ground decal.
- **Brigid**: flame ember stream (Points with additive sprite), burn patch ground tile, ultimate fire line (ribbon / stretched instanced quads along a path).
- **Susanoo**: lightning step flash (textured quad + screen-space bloom), Orochi's Wake channel (animated ribbon + spark Points burst).
- **Generic**: hit spark bursts (Points), pickup auras (Instanced billboard rings), destructible debris (InstancedMesh fragments with physics on low-detail path), shrine-break dust cloud (Points or sprite sheet).

**Building blocks needed** (estimate, authored once, reused everywhere):

1. `ParticlePool` — pre-allocated ring buffer of N instanced quads/points, one draw call per pool, uniforms per group. One `InstancedMesh` instance reduces 1000 particles to a single draw call — confirmed pattern in [three.js forum: InstancedMesh vs Points](https://discourse.threejs.org/t/better-performance-instanced-mesh-or-points/20293) (accessed 2026-04-24).
2. `Emitter` — spawn-rate, position zone, initial velocity jitter, lifetime; drains from a pool.
3. `ParticleMaterial` (ShaderMaterial on additive blending) — packed `aLifetime`, `aSeed`, `aColor` instance attributes; vertex shader computes position + size from `uTime`; fragment reads a sprite atlas.
4. `RibbonStrip` — 30–60 segment TubeGeometry updated from a rolling sample buffer, for Susanoo's lightning and Anansi's web strand.
5. `VFXController` (R3F component) — declarative `<HitSpark />`, `<EmberStream />`, `<WebStrand />` wrappers that drive the underlying pools through `useFrame`.

Author cost is roughly 4–6 engineer-days for blocks 1–3 (single, well-scoped primitives), 1–2 days for 4, 1 day for 5. Every incremental god kit is then shader-swap + atlas-swap, not a new engine. This cost is bounded and front-loaded; the payoff is ownership: we control sprite atlas packing (fits inside the 8 MB per-god budget in §15 R-15), we can auto-downgrade to `THREE.Points` on the Mobile Low preset (§7 perf notes), and we never ship dead three-nebula code for features we don't use.

## Alternatives considered

- **`three.quarks`** — Actively maintained TypeScript-first particle engine; latest `0.17.0` published 2025-01 (npm ~1.2 MB unpacked, ~38.5 KB min+gz), last commit 2026-02-05, has an R3F package (`quarks.r3f`) — [three.quarks repo](https://github.com/Alchemist0823/three.quarks) and [registry/latest](https://registry.npmjs.org/three.quarks/latest) (accessed 2026-04-24). Peer-dep is `three >=0.182.0` which will constrain our three version pin. Materially better than three-nebula on every axis except ecosystem recognition. **This is the sensible fallback if hand-rolling blocks Phase 2.**
- **`vfx-composer` (hmans/composer-suite)** — GPU/InstancedMesh-based, composable R3F library. Last publish `0.4.0` on 2022-10-28, so it's also effectively stalled — [npm registry](https://registry.npmjs.org/vfx-composer), [composer-suite repo](https://github.com/hmans/composer-suite/tree/main/packages/vfx-composer) (accessed 2026-04-24). Good architecture but the maintenance risk mirrors three-nebula; skip.
- **`@react-three/drei` `<Sparkles />` and `<Trail />`** — Built-in, zero-risk, but scoped to "floating glints" and "mesh-following trail" respectively; not a full emitter/pool system. Useful as a spot-tool inside the hand-rolled layer (e.g. pickup auras, projectile trails) — [drei Sparkles](https://drei.docs.pmnd.rs/staging/sparkles), [drei Trail](https://drei.docs.pmnd.rs/abstractions/trail) (accessed 2026-04-24).
- **`wawa-vfx`** — Small (~132 stars), TypeScript R3F VFX library with `VFXParticles` + `VFXEmitter` primitives — [wass08/wawa-vfx](https://github.com/wass08/wawa-vfx) (accessed 2026-04-24). Interesting but too new/small for a 14-day v1 dependency bet.

## Recommendation

**Hand-rolled `InstancedMesh` + `Points` layer**, with `@react-three/drei` `<Sparkles />` / `<Trail />` used tactically for pickup auras and projectile trails.

1. three-nebula fails our mobile floor: its GPURenderer does not run on iOS Safari ([issue #127](https://github.com/creativelifeform/three-nebula/issues/127)), leaving only the CPU SpriteRenderer, which is the opposite direction from the 30 FPS iPhone 12 / Pixel 6 target in §1 and the "no dynamic point lights, cascaded shadows off on mobile" budget in §7.
2. Package health is "inactive" per Snyk (48/100, no commits in 6 months, no release in 12) and the library has no TypeScript story we control, no official R3F wrapper, and a history of shipping three.js twice — taking this on means owning a dead dep on a 14-day timeline.
3. Our specific VFX list (web strand, fire line, lightning channel, ribbons, hit sparks) is a handful of well-understood primitives that an owned `ParticlePool` + `RibbonStrip` layer covers in ~5 building blocks. Hand-rolling is bounded effort, fits our per-god 8 MB asset budget (§15 R-15), and keeps the Accessibility reduced-motion hook (§13) a trivial uniform tweak rather than a vendored-library patch.

## Implementation note for Phase 2

Author the five blocks below inside `src/vfx/` (one PR each, small surface):

1. **`ParticlePool.ts`** — Backed by `THREE.InstancedMesh(quadGeometry, shaderMaterial, N)`; instance attributes `aBirthTime`, `aLifetime`, `aSeed`, `aColor`. Pool exposes `spawn(position, velocity, options)` that writes into a free slot; free slots tracked by a ring buffer cursor. Targets: 256 particles / pool on Mobile Low, 1024 on Desktop.
2. **`Emitter.ts`** — Declarative config (`rate`, `burst`, `positionZone`, `velocity`, `lifetime`, `colorOverLife`, `sizeOverLife`). Not a scene object — a plain object consumed by `useFrame` in the controller.
3. **`particle.glsl`** — Vertex shader evolves position from `aBirthTime + uTime` analytically (gravity + drag), so CPU never writes per-particle transforms. Fragment samples a 4×4 sprite atlas indexed by `aSeed`. Additive blend, depth-write off, depth-test on.
4. **`RibbonStrip.ts`** — Uses `THREE.BufferGeometry` with a rolling array of 30–60 sample points; rebuilds quad strip each frame (cheap at this size). Two variants: world-space trail (Susanoo lightning), local-space billboard (Anansi web strand, Brigid fire line).
5. **`VFXController.tsx`** — R3F wrappers (`<HitSpark />`, `<EmberStream />`, `<WebStrand />`, `<LightningArc />`) that allocate pool slots on mount, schedule emissions via `useFrame`, and honor the Reduced Motion setting (§13) by scaling `count` and `lifetime` uniforms to 50%/30%.

Mobile preset (`mobileLow` capability flag, set by the Safari-profiling pass noted in §15 R-14) halves pool sizes and swaps `InstancedMesh` for `THREE.Points` (single-quad billboard) where silhouette detail doesn't matter (hit sparks, embers, dust). Keep ribbons on `InstancedMesh` on all tiers — they carry silhouette.
