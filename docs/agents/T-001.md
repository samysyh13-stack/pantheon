# T-001 — Unified Input Manager

**Persona**: IN (Input Systems Engineer)
**Task**: Implement the unified input manager plus virtual joystick primitive, wire remap persistence into Settings, extend the store, and back it with unit tests.
**Status**: Complete — `npm run typecheck` passes, `npm run test` passes (47/47 including 18 new cases in `tests/unit/input.test.ts`).

## Summary

Delivered a deterministic, per-tick-sampled unified input layer covering keyboard+mouse, Gamepad API, and multi-touch, exposed via a single `InputManager` factory under `/src/game/systems/input`. Each input surface is isolated in its own module with a snapshot-only surface; the manager samples all sources each render frame, merges them into the authoritative `InputFrame` shape from ARCHITECTURE §12, and exposes both a push (`subscribe`) and pull (`snapshot`) API so the 60 Hz fixed-tick sim can read on demand without waiting on rAF (ADR-0006 compliance — no `Math.random`, no wall-clock inside the frame). The settings store gained a `controls` slice with keyboard and gamepad keymaps plus the touch scale/opacity; the old standalone `touch` field was folded into it with a v1→v2 persist migrator. A React `<Joystick />` primitive renders base + thumb into the existing HUD anchor slots via a subscribing hook, without touching HUD.tsx.

## Files created

- `/src/game/systems/input/types.ts` — `InputFrame`, `Action` enum, `InputSource`, `PlayerIndex`, `ALL_ACTIONS`, `createEmptyFrame` helper.
- `/src/game/systems/input/defaults.ts` — shipped keyboard (WASD / Mouse0 / Mouse2 / Space / ShiftLeft / Escape) and gamepad (standard-mapping RT / RB / Y / LT / Start) keymaps, plus axis and deadzone constants.
- `/src/game/systems/input/keyboard.ts` — window-scoped KeyboardEvent.code depressed-set tracker; resolves move vector + action booleans from an injected keymap. Exposes `setMouseButton(button, down)` so `mouse.ts` shares the same depressed set (Mouse0 / Mouse2 tokens).
- `/src/game/systems/input/mouse.ts` — mouse position + button tracker; `setAimReferencePoint(x, y)` injects the character's screen-space point and the tracker returns a normalized aim vector with a linearly ramped magnitude up to a 160px saturation radius. Suppresses the default context menu so RMB-as-ability fires.
- `/src/game/systems/input/gamepad.ts` — `navigator.getGamepads()` poll with `gamepadconnected` / `gamepaddisconnected` hot-plug; radial deadzone applied to both sticks; standard-mapping button reads.
- `/src/game/systems/input/touch.ts` — multi-touch tracker (3-concurrent cap), left-half binds to move joystick, right-half to aim joystick + button overlay hit-test; PointerEvent fallback for touch-capable browsers that don't fire TouchEvent; subscribe API for the React Joystick component.
- `/src/game/systems/input/virtualJoystick.ts` — pure thumb-dock state machine; dock = first-touch position, thumb output is the unit-clamped delta, release resets. Shared by both left and right sides.
- `/src/game/systems/input/haptic.ts` — `pulse(ms)` and `pulsePattern(pattern)` wrappers over `navigator.vibrate`; feature-detects and logs a single console.info on first call when missing (iOS Safari); `__resetHapticForTests` escape hatch.
- `/src/game/systems/input/manager.ts` — central `create({ playerCount })` factory; assigns sources per brief (solo = merged, multi = P0 keyboard+mouse, P1..n = gamepad slots then touch); rAF-driven internal loop; `subscribe` / `snapshot` / `setAimReferencePoint` / `dispose` public API.
- `/src/game/systems/input/index.ts` — public re-export barrel.
- `/src/ui/components/Joystick.tsx` — React component that resolves the HUD's existing `#joystick-left` / `#joystick-right` anchors, subscribes to the touch tracker, and renders base + thumb. Reads `controls.touchScale` and `controls.touchOpacity` from the store. HUD.tsx is untouched.
- `/tests/unit/input.test.ts` — 18 cases: `deadzone` math sanity, `ALL_ACTIONS`/`DEFAULT_KEYBOARD`/`DEFAULT_GAMEPAD` shape/value asserts, empty-frame zero-vector invariant, remap roundtrip + persistence-shape test, virtual joystick state machine, haptic iOS no-op fallback.

## Settings migration notes

The store's `Settings` interface gained `controls: ControlSettings` (keyboard + gamepad keymap records, `touchScale`, `touchOpacity`). The old `touch: TouchSettings` slice and its `setTouch` action were removed; the `controls` slice gained `setControls` / `setKeyboardBinding` / `setGamepadBinding` in their place.

The persist version bumped `1 → 2`. The migrator (`migrateV1ToV2` in `/src/state/store.ts`) copies `touch.scale` → `controls.touchScale`, `touch.opacity` → `controls.touchOpacity`, and seeds keymaps from the baked defaults. Any other v1 fields (graphics preset, audio, accessibility) are forwarded with defensive `...defaults` spreads so a partial v1 record still loads cleanly.

Keymap defaults are **duplicated** between `src/state/store.ts` (for the persist seed) and `src/game/systems/input/defaults.ts` (for the runtime layer). This is deliberate — importing the input defaults into the store would introduce a build-time cycle (`store -> input -> store`). The unit test asserts shape parity; the inline comment in `store.ts` flags the mirror.

## Known limitations / deferred to Phase 2

1. **Haptic hook-up** — `pulse()` is exported but no gameplay system calls it yet. Combat (T-1xx) wires it on hit confirms and button presses per DESIGN §4 "Combat feel stack".
2. **Multi-player source reassignment mid-match** — current logic resolves P0..Pn once at `create()` time. Hot-swapping a gamepad between P1 and P2 mid-match isn't supported. Acceptable for v1 (same-device local co-op is seated at match-start).
3. **Remap UI** — the store actions (`setKeyboardBinding` / `setGamepadBinding`) exist, but the Settings screen UX component to drive them is UX's responsibility (future UX task).
4. **Gamepad per-vendor overrides** — `gamepadButtonForAction()` in `defaults.ts` accepts a per-gamepad override map but no vendor detection is wired. Current Xbox / PS5 / generic controllers all honour the Standard Mapping; if a controller shows up that doesn't, wire detection off `gamepad.id`.
5. **Touch gesture library** — @use-gesture/react is in the package deps but the current implementation uses raw `TouchEvent` + `PointerEvent` which covers every case cleanly and with better perf. Kept the dep; left a comment in `touch.ts` if a future gesture (e.g., pinch-to-zoom map) needs it.
6. **Per-player aim reference** — `setAimReferencePoint(playerIndex, x, y)` is stored per-source but only forwarded into `mouse.ts` for the keyboard+mouse-owning player (P0 in multi, or solo). Touch aim is joystick-delta-driven and doesn't need a reference point; gamepad aim is absolute stick input.
7. **InputFrame emission rate** — callbacks fire per-rAF (up to display Hz). Sim consumers should prefer `snapshot()` from inside the fixed-tick loop rather than accumulating edge events across subscribe callbacks, which can fire at 120 Hz on high-refresh displays.

## How to verify

From the repo root:

```
npm run typecheck
npm run test -- tests/unit/input.test.ts
```

Both must pass. Full suite is also green: `npm run test` → 47 tests across 4 files.

Manual smoke-test in the dev server (`npm run dev`):

1. Desktop: load the page, click **Play**, the HUD stub appears. WASD shouldn't move anything yet (no character controller yet — Phase 2), but you can verify event wiring by opening DevTools, running:
   ```
   const { create } = await import('/src/game/systems/input/index.ts');
   const mgr = create({ playerCount: 1 });
   setInterval(() => console.log(mgr.snapshot(0)), 500);
   ```
   Hold W → `moveY: 1`; move the mouse → `aimX/aimY/aimMagnitude` populate; LMB / RMB / Space / Shift / Esc toggle the respective booleans.
2. Gamepad: plug in any Xbox / PS5 / generic controller and press any button to wake the browser's gamepad stack. The snapshot output should show `source: 'gamepad'` when the right stick out-magnitudes the mouse, and buttons should map per DEFAULT_GAMEPAD.
3. Touch (or DevTools device mode): place a finger on the left half → `moveX/moveY` populate; drag on the right half → `aimX/aimY/aimMagnitude` populate; lift → both zero. The `<Joystick />` primitive doesn't auto-mount in the current HUD yet (UX adds that in a later task) — you can verify the state via `touch.snapshot()`.
4. Remap persistence: open DevTools console, run
   ```
   const { useAppStore } = await import('/src/state/store.ts');
   useAppStore.getState().setKeyboardBinding('ultimate', 'KeyQ');
   ```
   Reload the page and inspect `localStorage.getItem('panthenon-state')` — the `controls.keyboard.ultimate` should be `"KeyQ"`.

## Questions for orchestrator

- Do we want the default touch-button overlay (basic / ability / ultimate / dodge) rendered inside `<Joystick side="right" />` or as a separate component? The brief says "right half binds to aim joystick + button overlay region"; I implemented the hit-test region API in `touch.ts` but deferred the overlay's visual component to UX (T-005 or later), who owns `/src/ui/menus` and HUD. `<Joystick />` today is stick-only; the overlay lives wherever UX wants to place it and feeds `touch.setButtonRegions()`.
- Touch-vs-mouse coexistence on hybrid devices (Surface, iPad with trackpad): the current code accepts both and the last-pressed-source-wins merge in `mergeSoloFrame` handles it, but I haven't canonicalized behaviour (e.g., "if a touch arrived within the last 500 ms, lock the HUD to touch mode"). Easy to add if playtest reveals confusion.
