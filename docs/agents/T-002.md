# T-002 — Rendering Pipeline & Shader Foundation

**Persona**: RS (Rendering & Shader Specialist)
**Task**: Stand up the preset-driven post-processing pipeline, stylized toon/rim materials, tiered environment, and WebGPU-opt-in R3F Canvas — all wired against /docs/ARCHITECTURE.md §9's per-preset effect composition and /docs/DESIGN_DOCUMENT.md §3's art direction.
**Status**: Complete. `npm run typecheck` passes clean; `npm run test -- tests/unit/rendering-presets.test.ts` passes 17/17; `npm run build` succeeds (full bundle inventory below).

## Summary

Replaced the Phase-0 Canvas placeholder with a full-fat render stack: the R3F `<Canvas>` now reads a resolved `PresetConfig` to configure `gl`, `dpr`, `shadows`, and the camera, then mounts a tiered `<GameEnvironment>` (procedural gradient sky for Low/Battery, drei `<Environment preset="warehouse">` for Medium+), a temporary demo scene (cube + ground) preserved for Phase-1 smoke testing, and — on the WebGL2 path — an in-house `<EffectStack>` that assembles a `postprocessing` `EffectComposer` honouring the §9 table (outline, bloom, color-grade LUT, SSAO on Ultra, ACES/Linear tone map). Outline is a custom depth-based Sobel `Effect` subclass (not the pmndrs selection-based `OutlineEffect`, which doesn't match the cel-shading brief). Materials — a banded-diffuse + rim toon on `MeshStandardMaterial` and a rim-only lightweight variant on `MeshBasicMaterial` — are built on `three-custom-shader-material` and re-exported from `/src/rendering/index.ts`. WebGPU is behind the `settings.renderer === 'webgpu'` opt-in with a `navigator.gpu` feature gate, dynamic-imported from `three/webgpu` (the brief's reference to `three-stdlib` is incorrect — three-stdlib does NOT re-export a WebGPURenderer; this is documented in the Canvas.tsx top-of-file comment).

## Files created / modified

- `/src/rendering/Canvas.tsx` — REPLACED. New Canvas owns preset-driven `gl`/`dpr`/`shadows`/`camera`, WebGPU opt-in via dynamic `three/webgpu` import, composer mount guard, demo scene + environment mount.
- `/src/rendering/presets/index.ts` — extended `PresetConfig` with `tonemap: 'aces' | 'linear'` and `environment: 'gradient' | 'hdri'`; populated the two new fields on all 5 tier rows. Existing fields untouched so the Phase-0 stub callers remain valid.
- `/src/rendering/postprocessing/outline.ts` — `createOutlineEffect(thickness | config)` returns a custom depth-Sobel `Effect` (attribute: DEPTH). 9-tap 3×3 kernel on `readDepth(uv)`, sky-mask for far-plane skip, three uniforms (`uThickness`, `uStrength`, `uColor`).
- `/src/rendering/postprocessing/bloom.ts` — `createBloomEffect({ threshold, intensity, radius })` wrapper around pmndrs `BloomEffect`. Mipmap blur, screen blend, luminance-threshold gating for emissive-only bloom.
- `/src/rendering/postprocessing/colorGrade.ts` — `createColorGradeEffect(lut?)`. When `lut` is provided, constructs `LUT3DEffect`; when null/undefined, returns a minimal passthrough `Effect` so the EffectPass ordering stays stable until real LUTs land.
- `/src/rendering/postprocessing/ssao.ts` — `createSSAOEffect(config?)` desktop-only wrapper around pmndrs `SSAOEffect`. 9 samples, 7 rings; world-distance cutoffs tuned for the 40 m arena.
- `/src/rendering/postprocessing/composer.tsx` — `<EffectStack preset={} accessibility={}>` React component. Builds `EffectComposer` + `RenderPass` + optional `NormalPass` + single collapsed `EffectPass(outline, bloom?, colorGrade?, ssao?, toneMapping)`. Drives via `useFrame` priority 1 (overrides R3F's default render). Disposes composer + effects on unmount or dependency change.
- `/src/rendering/materials/Toon.tsx` — `<ToonMaterial color rimColor rimIntensity rimPower bandCount metalness roughness />`. CSM on `MeshStandardMaterial`. Fragment quantizes diffuse luminance into `uBandCount` bands and adds a fresnel rim via `uRimColor * rim * uRimIntensity`.
- `/src/rendering/materials/Rim.tsx` — `<RimMaterial baseColor rimColor rimPower rimIntensity />`. Lightweight sibling on `MeshBasicMaterial` for projectiles/pickups — no banded diffuse, rim only.
- `/src/rendering/materials/index.ts` — re-exports `ToonMaterial`, `RimMaterial` and their prop types.
- `/src/rendering/environment.tsx` — `<GameEnvironment preset={} />`. Gradient hemisphere (BackSide `<mesh>` with a 2-color `<shaderMaterial>`) on Low/Battery; drei `<Environment preset="warehouse">` on Medium+ (ADR-0012 neutral overcast choice — warehouse is the lightest preset without a strong directional sun).
- `/src/rendering/index.ts` — public barrel: `GameCanvas`, `GameEnvironment`, `ToonMaterial`, `RimMaterial`, `PRESETS`, `resolvePreset`, `detectPreset`, plus type re-exports.
- `/tests/unit/rendering-presets.test.ts` — 17 cases covering: PRESETS table populated + shape consistent; outline-px / shadow-map-size / target-FPS per §9 rows; effect gating (ssao Ultra-only; bloom Medium+; color grade all-except-battery; tonemap ACES on M+/Linear on L/B); `resolvePreset('auto')` is one of the 5 concrete tiers; `detectPreset()` stubbing navigator returns expected tier for desktop-hi/mid/low, mobile-hi/mid/low.

The `post-*.js` chunk is the single largest — it contains the entire `postprocessing` library plus the runtime side of `three-custom-shader-material`. Gzipped at 265 kB, it's within the §4 500 kB engine budget, but we should flag it for the Phase-2 PO (Performance Optimizer) pass: tree-shaking is not removing unused `postprocessing` Effects we don't use (DotScreen, ShockWave, Glitch, GodRays, DepthOfField, etc.) because `postprocessing` publishes ESM with `sideEffects: false` but its internal imports pull the effect classes eagerly.

WebGPU payload (`three/webgpu`, ~637 kB gzip'd min) is NOT in any of the above chunks — it's dynamic-imported and only loads when `settings.renderer === 'webgpu'` AND `navigator.gpu` exists. Verified by grepping the `three-*.js` chunk: only 15 incidental WebGPU mentions (constants), none of the renderer code.

## Known gaps (deferred to Phase 2)

1. **Real HDRI for Sacred Grove** — drei's `Environment preset="warehouse"` is the Medium+ placeholder. T-104 (arena polish) authors the arena-specific neutral-overcast HDRI at KTX2, served from R2, and swaps this file's `<Environment preset="warehouse">` for an asset-path variant.
2. **LUT textures** — `createColorGradeEffect()` accepts a `Texture | null` but we ship `null` today (passthrough `Effect`). T-104 / CD (Creative Director) should author at least two LUTs: a warm "duel" grade and a cool "totem rush" grade. Loader wiring lives in the composer — pass the loaded Texture through on `<EffectStack lut={...} />` (add the prop when textures exist).
3. **WebGPU composer parity** — pmndrs `postprocessing` is WebGL2-only at time of writing. The WebGPU path currently bypasses the composer entirely (renderer native tone mapping applies). If WebGPU adoption grows past Phase 2, port to three's node-based `PostProcessing` (from `three/webgpu`) and add a parallel composer path. Flagged explicitly in Canvas.tsx's top-of-file doc block.
4. **Emissive gating for bloom** — currently a luminance threshold (0.9). /docs/DESIGN_DOCUMENT.md §3 specifies "bloom only on emissive" — the threshold approximation works when character/arena albedo stays below 0.9 luma. If characters end up with high-albedo stylized highlights (e.g., pure-white highlights on Susanoo's blade), migrate to `SelectiveBloomEffect` with an emissive layer, at the cost of adding a Selection registration point to every emissive mesh.
5. **Dithering + colorspace output** — the composer uses pmndrs defaults which encode to sRGB on the final pass. If banding appears in the gradient sky on mobile, enable EffectPass.dithering in the composer (one flag). Leaving off for now to avoid the minor perf hit on mobile.
6. **Reduced-motion wiring** — bloom baseline softens (0.9 → 0.55). Hit-flash suppression and camera-shake clamping live in combat (Phase 2, T-1xx). No action needed in rendering until combat lands.
7. **Colorblind palette swap** — brief explicitly reserves for the UX layer. No rendering-side work required.
8. **Shader-variant accounting automation** — the current "count by reading the source tree" is manual. A Phase-2 PO task could wire a build-time ESLint rule that tallies `CustomShaderMaterial`, `ShaderMaterial`, and `new Effect(...)` constructions against a threshold defined in /docs/ARCHITECTURE.md.

## Questions for orchestrator

1. **Pre-existing `/src/game/systems/input/haptic.ts` TS 2430 error.** On a cold typecheck (fresh `tsc` cache) the `NavigatorWithVibrate` interface declaration errors because TS 6.0's `lib.dom` types tightened `Navigator.vibrate`'s parameter to `VibratePattern` (which is `number | Iterable<number>`) rather than `number | number[]`. On a warm typecheck (after my changes compiled at least once) the build info apparently caches through it and `npm run typecheck` + `npm run build` both pass. I did NOT modify haptic.ts — strictly off-limits per the T-002 brief (`STRICTLY DO NOT TOUCH: /src/game/**`). If this error resurfaces on the next CI run, the two-character fix is changing `number | number[]` to `VibratePattern` in haptic.ts line 9. Flagging for orchestrator integration. (Worth noting this is a pre-existing T-001 regression caused by a TS/@types/dom bump — not my code.)

2. **`@react-three/postprocessing` install?** The brief listed "if you hit a blocker needing @react-three/postprocessing, list in Questions". I did NOT need it — the in-house `<EffectStack>` in composer.tsx covers the full pipeline with direct `postprocessing` imports, stays TS-strict-clean, and keeps `exactOptionalPropertyTypes` happy. We're fine without the extra dep. If a future task wants R3F-JSX-ish `<EffectComposer><Bloom/></EffectComposer>` syntax, we can revisit — but for T-002 it would only add bundle weight without capability gain.

3. **Bundle chunking strategy for `postprocessing`.** The `post-*.js` chunk at 265 kB gzip'd is heavy for what we use (~5 of the 30+ effects). The `manualChunks` rule in vite.config.ts lumps all postprocessing + CSM into one chunk. Worth a Phase-2 PO investigation: should we split per-effect (e.g., `postprocessing/bloom` as its own chunk) or stick with the all-in-one post chunk? Recommending leaving it as-is until actual p95 TTI data comes in — premature optimization otherwise.

4. **HDRI asset path convention.** drei's `Environment preset="warehouse"` hits a CDN URL baked into drei. On mobile with spotty connectivity, that first-paint will block on a ~500 kB HDRI download. Mitigation options (pick one for T-104):
   - (a) `Environment files="/hdri/sacred-grove.ktx2"` with the asset shipped in `public/hdri/` — adds to the static bundle size but removes the external dependency.
   - (b) Preload via a drei `useEnvironment.preload({ files: ... })` call before the match starts, so the main menu's "Play" click doesn't stall on HDRI load.
   Recommendation: (a) with a KTX2-compressed HDRI around ~200 kB. Log an ADR if we go (b).

5. **Tone mapping ownership — composer vs renderer.** I set `gl.toneMapping = NoToneMapping` via `onCreated` and set `<Canvas flat>` so the composer owns tone mapping. This is the pmndrs recommendation. If a future task wants per-mesh override (e.g., HUD billboards rendering in sRGB-pass-through), note that the composer's ToneMappingEffect runs last and will re-tonemap those billboards. The HUD is DOM-overlay today so this isn't a concern yet.
