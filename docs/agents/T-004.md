**Persona**: AE (Animation & Character Engineer)
**Task**: Implement the character entity stack for Phase 1 — a Rapier kinematic character controller, a pure-TS animation FSM, an R3F `<Character>` component, an isometric tracking camera, a demo subtree, a minimal Anansi wrapper, and unit tests for the FSM.

## Summary

Delivered seven source files under `/src/game/entities/character/` plus a thin god wrapper under `/src/game/gods/anansi/`:

- `types.ts` — `CharacterConfig`, `CharacterState`, `CharacterStats`. Pure types, no three.js / Rapier imports, so the FSM tests can import them without booting the WASM physics runtime.
- `animationFSM.ts` — deterministic 6-state FSM with interrupt rules, time budgets, and auto-transition on expiry. Exports `ATTACK_DURATION_MS = 450`, `DODGE_DURATION_MS = 300`, `HIT_DURATION_MS = 250` as the single source of truth.
- `controller.ts` — `createCharacterController(deps)` factory that owns a Rapier `KinematicCharacterController`, applies slope/step handling, computes per-tick velocity via critically-damped acceleration (with separate accel / decel lambdas), implements the dodge roll + i-frame predicate + 3 s cooldown, and exposes edge-detection methods for FSM drive.
- `Character.tsx` — R3F component that mounts a `kinematicPosition` rigid body + capsule collider (radius 0.4, half-height 0.9), binds the controller/FSM to the input source per tick via `useFrame`, and exposes an imperative handle (`CharacterHandle`) via an `onHandleReady` callback rather than `forwardRef` (React 19 ref-as-prop semantics were simpler to reason about this way).
- `Camera.tsx` — `<TrackingCamera target={ref} />` that drives the R3F default camera to a (0, 10, 14) offset, look-at (0, 1, 0), critically-damped at `lambda = 6.0` on both position and look-at. Snaps on first mount to avoid the race-to-origin glitch; interpolates thereafter.
- `Demo.tsx` — `<CharacterDemo />` renders one ground plane + one Anansi-statted character + tracking camera. Must be mounted inside an R3F `<Canvas>` AND a `<Physics>` provider (orchestrator wires `<Physics>` in the Canvas.tsx swap — see Question 2).
- `index.ts` — barrel re-exports for the character folder.
- `/src/game/gods/anansi/Anansi.tsx` — 30-line wrapper forwarding HP 320 / moveSpeed 5 / gold #D4A24A to `<Character>`. Full kit lands in T-100 (CB, Phase 2).
- `/tests/unit/characterFSM.test.ts` — 24 cases covering every transition required by the brief plus the duration-constant parity, timeInState accumulation, defensive non-positive-dt handling, and the dead-is-terminal invariant.

```
         │    ▲         │        │                  │                   │

    > 0.1   ≤ 0.1       │   ┌─────┘                │                     │
         │    │         │   ▼                      ▼                     │
         ▼    │         │   idle                  idle                  idle
      ┌─────────────┐   │

      └─────────────┘

      ─── external transitions ───        ─── auto time-elapse ───
      any non-dead ──► dead   (terminal; no exit)
      any (except dead / hit-during-hit) ──► hit

      ─── interrupt rules (codified in canTransition()) ───
      attacking → dodging          OK     (dodge-cancel)
      attacking → idle / running   BLOCKED (no player self-cancel;
                                            wait out the 450 ms budget)
      dodging   → attacking         BLOCKED (dodge commits to i-frames)
      dodging   → dodging           BLOCKED (cooldown is controller-owned)
      hit       → hit               BLOCKED (stun-lock clip-guard)
      dead      → anything          BLOCKED (terminal)

      ─── i-frame predicate ───
      isInvulnerable() ≡ (current === 'dodging' && timeInState < 300 ms)
      (implemented on the Controller, not the FSM; the Controller owns
      the dodgeTimerMs, which drives both the FSM transition timing and
      the i-frame predicate — single source of truth)
```

## Controller ↔ InputFrame loop per tick

Every tick (dt = 1/60 s, per `src/game/engine/loop.ts::TICK_DT`) the following runs inside `Character.tsx`'s `useFrame` callback:

```
 1. Snapshot input:
      input = inputSource.snapshot(playerIndex)

 2. Update mouse aim reference (feature-detected; real InputManager only):
      project(characterWorldPos) → screenX, screenY
      inputSource.setAimReferencePoint(playerIndex, screenX, screenY)
    So mouse aim is relative to the character's screen-projected position,
    not the window origin — required because the camera can pan.

 3. Controller.update(input, dt):
      a. Rising-edge detection: dodgeEdge, basicEdge
      b. Dodge activation (if dodgeEdge AND cooldownReady AND not-dodging):
           - Latch dodge direction from move input (or back-dodge if neutral)
           - dodgeTimerMs   = 300
           - dodgeCooldownMs = 3000
           - pendingDodgeEdge = true  (consumed by FSM driver below)
      c. Decrement dodgeTimerMs / dodgeCooldownMs by dt*1000
      d. Compute target planar velocity:
           - During dodge: targetV = dashSpeed × latchedDir
           - Otherwise   : targetV = moveSpeed × normalize(input.moveX, -input.moveY)
                           (Y flip: stick-up → world -Z "forward")
      e. Critically-damped lerp current → target:
           - ACCEL_LAMBDA = 14.0 when speeding up
           - DECEL_LAMBDA =  9.0 when slowing down
           - (dodge doubles ACCEL_LAMBDA for immediate snap to dash speed)
      f. Gravity accumulator: velY += -20.0 × dt (capped to -40 m/s)
      g. Feed desired = (velX*dt, velY*dt, velZ*dt) to Rapier's
         KinematicCharacterController.computeColliderMovement(collider, desired)
      h. Apply computedMovement() via rigidBody.setNextKinematicTranslation
      i. Zero velY when computedGrounded() && velY < 0 (prevent gravity creep)

 4. Drive the FSM from controller edges and derived state:
      a. if ctrl.consumeDodgeEdge():       fsm.transition('dodging')
      b. if ctrl.consumeBasicAttackEdge(): fsm.transition('attacking')
      c. velocity threshold:
           idle,    speed > 0.1 m/s  → fsm.transition('running')
           running, speed ≤ 0.1 m/s  → fsm.transition('idle')

 5. fsm.tick(dt) — accumulates timeInState and auto-fires time-elapse
    transitions (attacking → idle at 450 ms, dodging → idle at 300 ms,
    hit → idle at 250 ms). Never auto-transitions from `dead`.
```

Key binding invariants:

- **Edges, not levels**: basicAttack and dodge both become FSM transitions only on the rising edge. The controller owns edge detection so the FSM stays pure and the controller's cooldown bookkeeping isn't duplicated.
- **FSM never blocks the controller**: the controller doesn't read FSM state. This keeps physics deterministic even if a test or debugging tool force-sets a bad FSM state. Hit and death are external-only transitions; everything else is a request the FSM can veto via `canTransition`.
- **Dodge ownership**: the Controller is the single source of truth for dodge timing (`dodgeTimerMs`). The FSM mirrors it (transition + auto-exit at 300 ms) but the i-frame predicate lives on the Controller because that's where the timer lives. Exposed on `Character`'s handle via `isInvulnerable()`.
- **Cooldown, not FSM**: the 3 s dodge cooldown lives on the Controller only. The FSM doesn't know about it — it just knows the `dodging` state lasts 300 ms. The Controller gates the dodge-edge consumption.

## Camera tracking parameters

Exports from `src/game/entities/character/Camera.tsx`:

| Constant | Value | Rationale |
|---|---|---|
| `CAMERA_OFFSET` | `[0, 10, 14]` | Height 10 m, 14 m back (+Z). Pitch angle `atan2(10, 14)` ≈ **35.5°** — slightly shallower than true iso (30°/60°) to keep arena floor visible without squashing character silhouettes. Matches AE brief's "45° angle, distance ~14 m, height ~10 m". |
| `CAMERA_LOOK_AT_OFFSET` | `[0, 1.0, 0]` | Aim 1 m above the character's origin (roughly mid-chest on a 2.6 m-tall capsule) for natural framing. |
| `CAMERA_LAMBDA` | `6.0` | Critically-damped `damp()` from `/src/utils/math.ts`. 99% approach in ~0.76 s — smooth but responsive; empirically the sweet spot between "twitchy on mobile" (lambda > 10) and "chases a laggy character" (lambda < 4). Same lambda applied to position and look-at to prevent framing drift. |
| `CAMERA_FOV_DEG` | `45` | Matches Canvas.tsx's default. Perspective rather than orthographic — the Sacred Grove standing stones (§7) read better with vertical foreshortening. |
| `CAMERA_NEAR` / `CAMERA_FAR` | `0.1` / `200` | Matches Canvas.tsx. 40 m arena fits with headroom. |

On first-mount the camera **snaps** to the target's offset (no interpolation) so the "camera racing from origin to the player" glitch doesn't appear; every subsequent frame uses `damp()`. The `damp` helper is critically-damped by construction (`damp(c, t, λ, dt) = lerp(c, t, 1 - exp(-λ*dt))`) so there is no overshoot — it's a low-pass filter, not a spring-mass solver — satisfying the brief's "no snap, no overshoot" constraint.

The camera does not mount a new perspective camera; instead it drives the R3F default camera (`useThree().camera`) in place. This keeps the Canvas.tsx integration stable — Canvas.tsx already constructs its own camera, and our tracking logic just repurposes that instance.

## Known gaps / deferred to Phase 2

1. **No animation clips yet** — the FSM is state-only; no glTF mixer, no crossfades, no blend times. The AE-in-full brief in `/docs/AGENTS.md` §3 specifies Mixamo humanoid retargeted + Blender polish as Phase 2 / T-101 scope. The FSM states map 1:1 to animation slots today but drive nothing visual beyond the capsule geometry. T-101 will wire `AnimationMixer` behind the `current` state + `timeInState` the FSM already exposes.

2. **Hit application stubbed** — `applyHit(dmg)` transitions to `hit` and returns `true`/`false` on invulnerability, but no HP arithmetic, no damage number spawn, no hitstop, no hit-flash, no camera shake. DESIGN §8's full hit-confirm pipeline is T-100/T-102 territory. The signature is final — Phase 2 just adds the bookkeeping behind it.

3. **No character facing / rotation** — the rigid body has `enabledRotations={[false, false, false]}` so the capsule doesn't tumble; the render mesh does not yaw toward aim or movement direction. Aim-following yaw lands with T-101's rigged model (capsule has no "front").

4. **Controller RNG carried but unused** — `createRng(config.seed)` is threaded through for Phase 2 features (footstep surface picker in DESIGN §11, hit-flash intensity variation). Doesn't affect movement.

5. **No second LOD** — per ARCHITECTURE §9 "Characters have two LODs (near / far) swapped at 18 m camera distance". The capsule is a placeholder; LODs attach in T-101 when the rigged models exist.

6. **Dodge direction when neutral is a guess** — if the player presses dodge with zero move input AND zero aim, the dodge defaults to `+Z` (screen-down in iso view). Not bad, but worth a playtest opinion in Phase 2 QA (T-107). Alternatives: back-dodge always (aim-opposite), no-op the dodge edge (feels unresponsive), or mirror the most-recent non-zero input vector (tricky determinism).

7. **Aim-reference-point wiring** — `Character.tsx` feature-detects `setAimReferencePoint` on the input source and calls it each frame with the character's screen-projected world position. The real `InputManager` supports this; mocks and tests don't need to. The type cast (`as unknown as { setAimReferencePoint: ... }`) is a known hole in the strict type graph — if a future input-source shape changes, this will fail at runtime instead of compile-time. Acceptable given the shape is stable (T-001 complete).

8. **Controller determinism across identical inputs is structural but untested** — by construction the controller is deterministic (no Math.random, no wall-clock, fixed dt). An integration test that spawns two controllers with identical seeds and identical input streams, then asserts matching position histories, is a Phase 2 / QA task (T-107 / T-204). Not in T-004 scope.

9. **Rapier determinism beyond our controls** — Rapier's `@dimforge/rapier3d-compat` 0.19.2 (via react-three-rapier 2.2.0) defaults its timestep to `1/60` (`Physics` prop `timeStep`); we confirmed the controller works with the default. DESIGN §7/ARCHITECTURE §7 mention pinning CCD off for non-projectile dynamics and disabling sleeping on tracked bodies. The character is kinematicPosition (CCD doesn't apply) and `canSleep={false}` on the RigidBody. No Rapier-determinism blocker encountered — see Question 3 for the integration-commit wiring of `<Physics timeStep={1/60}>`.

10. **No `useBeforePhysicsStep` integration** — the brief ties the controller to `useFrame`, which runs before R3F's render. `@react-three/rapier` uses `useFrame` internally too (with priority) so physics step happens after our `useFrame` callback — that's correct: we set `setNextKinematicTranslation`, then Rapier steps and integrates kinematic velocities. If Phase 2 needs finer step-ordering, `useBeforePhysicsStep(cb)` is available from the rapier hooks; today we don't need it.

## Questions for orchestrator

1. **Integration wiring for `<Physics>`.** `Character.tsx` calls `useRapier()` and uses `<RigidBody>` + `<CapsuleCollider>`, all of which require a `<Physics>` provider ancestor. Canvas.tsx (per the brief) is off-limits to me. When you swap Canvas.tsx's `<DemoScene>` for `<CharacterDemo>` (or a direct `<Character>`), you'll need to wrap with `<Physics timeStep={1/60}>` — probably inside the `<Suspense>` alongside `<GameEnvironment>`. The character-demo subtree does NOT include `<Physics>` itself because I was uncertain whether Canvas.tsx would want a single top-level `<Physics>` (shared by arena colliders, projectiles, breakables) or one per scene. Recommending: one top-level `<Physics timeStep={1/60}>` inside the existing `<Suspense>` in Canvas.tsx, wrapping everything from `<GameEnvironment>` through the demo scene.

2. **Rapier deterministic mode — anything extra needed?** The brief says "document and ask if Rapier's deterministic mode hits a blocker (e.g., requires specific init call)". For our setup — kinematicPosition characters, `<Physics timeStep={1/60}>`, the character controller created via `world.createCharacterController(0.02)` — no special init was needed. `@dimforge/rapier3d-compat`'s WASM boot happens automatically inside `<Physics>`. ADR-0006's invariants (no Math.random, no wall-clock in sim, fixed dt) are honored by our code; Rapier itself is deterministic given identical inputs + identical timestep + identical initial conditions per dimforge's docs. If you want an integration smoke-test that runs two physics worlds with identical seeds + inputs and asserts state parity, that's a Phase 2 QA task — not in T-004 scope.

3. **Aim reference-point feature detection.** Our `Character.tsx` does:
   ```ts
   const hasAimRef = useMemo(() => 'setAimReferencePoint' in (inputSource as object), [inputSource]);
   ```
   This works because the InputSource interface only requires `snapshot`, but the real InputManager also implements `setAimReferencePoint`. If you ever pass a mocked input source in a non-test path and it happens to expose a `setAimReferencePoint` that doesn't match the real manager's signature, strange aim behavior results. Optional tightening for the orchestrator: add `setAimReferencePoint` as an optional method to `InputSource` and call it via `inputSource.setAimReferencePoint?.(...)` directly. Not critical — the current approach is safer for test mocks.

4. **Anansi wrapper scope — is the one-line `color="#d4a24a"` enough?** The `/docs/gods/ANANSI.md` in the repo is a stub, and T-100 (CB, Phase 2) owns the full kit. My `Anansi.tsx` only wires HP/moveSpeed/color; there's no rig, no projectile, no clone, no ultimate. Confirming that's the expected T-004-shape so you don't look at this file in integration review and think it's incomplete — it is deliberately minimal per the brief: "minimal, just forwards stats".

5. **One observation during integration review: `src/app/App.tsx` still renders `<MainMenu onPlay={...} />`** — the current T-005 `MainMenu` ignores `onPlay` and drives screen state via the Zustand store. Typecheck passes (React FC type allows extra props through), so it's not a blocker, but the `onPlay` prop is now dead code and `screenState` in `App.tsx` is disconnected from the store's `match.screen` selector the menus write to. Not T-004 scope to fix, but flagging for the integration commit — once you swap `<DemoScene>` for `<CharacterDemo>`, you'll likely want App.tsx to read `useAppStore(s => s.match.screen === 'match')` rather than its own `screenState` local.
