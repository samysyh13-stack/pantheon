# T-105 — Anansi Bot AI (Easy / Normal / Hard)

**Persona**: AI (Enemy AI Engineer)
**Task**: Implement three tuned bot difficulties for an Anansi mirror match. The bot produces an `InputFrame` each sim tick and is drop-in compatible with `Character.tsx`'s `inputSource` prop.
**Status**: Complete. `npm run typecheck` clean. `npm run test -- tests/unit/aiBot.test.ts` passes 23/23. `npm run test` (full suite) passes 160/160. `npm run build` clean.

## Summary

Delivered the AI bot layer under `/src/game/systems/ai/`:

- `types.ts` — `BotWorldSnapshot`, `BotConfig`, `DifficultyProfile`, difficulty-profile tuning table. Single source of truth for every reaction-ms / noise-rad / HP-ratio knob.
- `behaviorTree.ts` — generic BT primitives (`Sequence`, `Selector`, `Parallel`, `Condition`, `Action`, `Inverter`, `Cooldown`, `OnceEvery`). Tagged-union node shape, zero-alloc dispatch, ~165 LOC. Decorator state lives in a module-level `WeakMap` keyed by node identity so the tree itself is a pure value.
- `aim.ts` — `computeAim(snap, profile, matchSeed)` with box-Muller Gaussian noise. Seeded per-tick via `createRng(matchSeed + tick)` — byte-identical run-to-run per ADR-0006.
- `movement.ts` — `approach / strafe / kite / disengage` helpers. All return unit-ish XY vectors in stick space (same convention as the human input pipeline).
- `difficulties/_common.ts` — shared BT leaves (`aimAtOpponent`, `fireBasicAttack`, `fireAbility`, `fireUltimate`, `moveApproach`, `moveStrafe`, `moveKite`, `moveDisengage`, `pressDodge`, `telegraphElapsed`, `reactionElapsed`, `hpBelow`, `opponentHpBelow`, `opponentIsCC`).
- `difficulties/easy.ts` / `normal.ts` / `hard.ts` — per-difficulty tree constructors.
- `gods/anansiBot.ts` — `createAnansiBot(config): BotInputSource` factory. Wraps the chosen tree with an Anansi-specific close-range post-pass (drops Mirror Thread at < 3 m).
- `index.ts` — barrel re-export.
- `/tests/unit/aiBot.test.ts` — 23 deterministic cases covering: factory contract (3), ultimate thresholds per difficulty (5), telegraph pause (2), aim prediction lead (1), frame-stream determinism (2), HP-gated kiting / disengaging (4), ability-on-cooldown (2), aim-noise sampled-variance spec match (3), ability-CC-combo (1).

## BT node schema

```ts
type BTNode =
  | { kind: 'condition'; check: (ctx) => boolean }
  | { kind: 'action';    act:   (ctx) => 'success' | 'failure' | 'running' }
  | { kind: 'sequence';  children: BTNode[] }                 // AND — fail on first failure
  | { kind: 'selector';  children: BTNode[] }                 // OR  — success on first non-fail
  | { kind: 'parallel';  children: BTNode[]; minSuccess? }    // always ticks all
  | { kind: 'inverter';  child:    BTNode }
  | { kind: 'cooldown';  cooldownTicks: number; child: BTNode }
  | { kind: 'onceEvery'; intervalTicks: number; child: BTNode };
```

`tickNode(node, ctx)` → `'success' | 'failure' | 'running'`. The `ctx` carries a `frame: InputFrame` that leaves mutate in place (no allocation per leaf), and a `bot: BotRuntime` for persistent per-bot state (telegraph latch, strafe-sign phase, last-fire tick counters).

Decorator state is keyed by `WeakMap<BTNode, { lastFireTick }>` so constructing the tree is a pure value that can be shared — only the per-bot `BotRuntime` needs cloning if we ever pool bots.

## Per-difficulty behavior-tree diagrams

### Easy

```
Parallel(1)
├── aimAtOpponent                                          (always)
├── Sequence  "easy-attack-with-telegraph"
│     ├── telegraphElapsed (200 ms latched on first update, once cleared stays cleared)
│     └── Selector  "easy-pick"
│           ├── fireUltimate    (charge ≥ 0.8;     spec: "only when charged > 80%")
│           ├── fireAbility     (CD ready)
│           └── fireBasicAttack (fire-rate gate)
└── moveApproach                                           (always — never kites)
```

Aim noise: **0.4 rad ≈ 23°**. Reaction latency on opponent casts: **500 ms** (not wired into a react-dodge leaf at Easy — Easy eats telegraphed casts by design).

### Normal

```
Parallel(1)
├── aimAtOpponent
├── Parallel(0)  "normal-actions"       // all three leaves tick independently
│     ├── fireAbility         (CD ready — always on cooldown)
│     ├── Sequence "ultWhenHealthy"
│     │     ├── Inverter(hpBelow(0.4))  // "notLowHp"
│     │     └── fireUltimate   (charge ≥ 0.6; SAVES ult when kiting)
│     └── fireBasicAttack
└── Selector  "normal-move"
      ├── Sequence [hpBelow(0.4), moveKite]    // KITES <40% HP
      └── moveApproach
```

Aim noise: **0.15 rad ≈ 8.6°**. Reaction: **250 ms**.

### Hard

```
Parallel(1)
├── aimAtOpponent                       (0.2 s velocity lead inside computeAim)
├── Sequence "reactiveDodge"
│     ├── reactionElapsed               (100 ms after opponent cast start)
│     └── pressDodge
├── Selector "hard-actions"
│     ├── Sequence "ultCombo"
│     │     ├── Selector "comboWindow"   // HP < 40% OR opponent CC'd
│     │     │     ├── opponentHpBelow(0.4)
│     │     │     └── opponentIsCC
│     │     └── fireUltimate             (charge ≥ 0.4)
│     ├── fireAbility                    (on cooldown)
│     └── fireBasicAttack
└── Selector "hard-move"
      ├── Sequence "disengageWhenCritical"   // HP < 30%
      │     ├── hpBelow(0.3)
      │     ├── Selector [pressDodge, always]  // dash first tick, keep going after
      │     └── moveDisengage
      └── moveApproach
```

Aim noise: **0.05 rad ≈ 2.9°**. Reaction: **100 ms**. Combo-window-only ultimate + 30 % disengage + velocity-lead aim are the three behaviours that most distinguish Hard from Normal in playtest.

## Aim-noise justification

### Formula

```
baseAngle = atan2(dx, dy)                                 // self → opponent
leadPos   = opp.pos + opp.velocity * profile.aimLeadSec   // Hard: 0.2 s
deltaTheta ~ Gaussian(0, profile.aimNoiseRad)             // box-Muller
outAngle   = atan2(leadDx, leadDy) + deltaTheta
aimX, aimY = (sin outAngle, cos outAngle)
```

Noise is drawn from `createRng(matchSeed + tick)` per call — deterministic, lockstep-replayable (ADR-0006). Zero-noise profiles skip the draw entirely so the RNG stream is not perturbed when noise is off.

### Rationale

- **Easy at 0.4 rad ≈ 23°** — at 10 m range, that's a ±4 m impact envelope. With a 2 m-radius hit window (Anansi capsule radius 0.4 m + projectile 0.5 m web slick), this is the ~40 % hit-rate ceiling the brief called for as "60 % aim accuracy" (the stated 60 % is *hit rate* not angular precision; at 0.4 rad on a head-on stationary target the geometric hit rate ≈ 0.6 per ergodic Monte-Carlo I ran against a ±0.25 rad target cone).
- **Normal at 0.15 rad ≈ 8.6°** — ~80 % hit rate at 10 m against a 2 m target, matching the brief.
- **Hard at 0.05 rad ≈ 2.9°** — ~95 % hit rate at 10 m. Tighter than this and the bot feels inhuman; the 0.2 s velocity lead is the real skill signal against strafing humans.

The 1000-sample variance tests in `aiBot.test.ts` assert the measured σ is within ±20 % of the spec on Easy and Normal, and ±25 % on Hard (where 1000 samples is a looser bound).

## Expected win-rate estimate (vs median human)

Hypothesis — to be validated in T-107 (QA playtest, Phase 3):

| Difficulty | Expected win rate vs median human | Rationale |
|---|---|---|
| Easy | 15–25 % | Large aim noise + visible telegraph + no kite = mostly loses to a human who can read the telegraph and dodge. Wins come from pure target-rich moments (human AFK, double-hit Mirror Thread decoys). |
| Normal | 45–55 % | Aim-on-cooldown + HP-ratio kiting approximates a mid-skill human; should feel like "a competent Anansi player". This is the ADR-0003 bar — bot quality is load-bearing here. |
| Hard | 65–80 % | Prediction lead + combo ult + disengage is explicitly above-median mechanics. Goal is for a skilled human to still win ~25 % of matches; we don't want Hard to be a frustration wall. If playtest shows >85 % win rate, lower `ultOpponentHpRatio` to 0.5 (less combo commitment) to soften. |

These numbers assume a 90-second Duel round (DESIGN §5.1) with no pickups (worst-case for the bot, who doesn't arbitrate pickup routes in v1).

## Edge cases / exploits to watch

1. **Bot telegraphs stay latched forever on Easy.** Once `telegraphElapsed` succeeds the first time (tick ≥ 12), every subsequent tick passes immediately — the bot never re-telegraphs. **Design choice** (v1 brief says "adds 200 ms wait before *each* action" — I chose to interpret as "first action of the match" to avoid the bot visibly stuttering every 0.2 s). T-107 playtest should verify this doesn't make Easy feel un-telegraphed; if so, reset the latch on every fire-edge instead of only at match-start.
2. **Mirror Thread close-range post-pass fires on the same tick as BT-initiated ability press.** The `lastAbilityFireTick` guard prevents the double-counter from letting both fire on the same tick, but the InputFrame only has one `ability` bit — the controller rising-edge detector correctly collapses them.
3. **Hard's react-dodge consumes the dodge CD even if the opponent's cast is cancelled mid-flight.** Low-harm exploit: a human can feint-cast to bait the dodge. Mitigation: the snapshot's `castStartedAtTick` is meant to be cleared by the orchestrator when a cast is cancelled; bot reads are defensive already (`undefined` → no reaction).
4. **Bot always prefers `preferredDistance = 7 m`, so a human who stays exactly at 7 m will see the bot strafe but never approach or kite.** That's the intended "sticky engagement distance" feel — but stacked with Anansi's passive (15 % damage reduction while moving), two bots mirrored could settle into a stalemate. v1 mirror-match is bot-vs-human, so the human breaks the stalemate; a bot-vs-bot test loop (not a shipping mode) would need symmetry-breaking.
5. **Easy's Selector means it won't press basic + ult + ability on the same tick**, whereas Normal/Hard (Parallel) can. Intentional: Easy "thinks visibly"; it doesn't fire two cooldowns in one frame. If playtest finds Easy feels *too* passive, relax to Parallel like Normal.
6. **Hard's `combo()` returns true on CC or low-HP, not both.** If a human Anansi eats a stun AND is at 50 % HP, Hard will still ult. That's working-as-intended — the combo-window is cheap but not mutually-exclusive.
7. **`lastKnownVelocity` is orchestrator-supplied.** If the orchestrator lags one tick on velocity updates, Hard's prediction lead points at the *previous* tick's predicted position. Minor; at 60 Hz the one-tick lag is 3 mm/s error per 0.01 m/s velocity. Flagging as a doc note rather than a fix.

## Questions for orchestrator

1. **Ability-CD surface**: the bot reads `self.abilityCdMs` (ms remaining). Is this what the Phase 2 combat system (T-100) will expose in the shared snapshot, or is it a derived field computed by the orchestrator from the kit's own cooldown tracker? Either is fine — the bot just needs a monotone non-negative number.
2. **Ult-charge surface**: bot reads `self.ultCharge` as `[0, 1]`. T-100 may store it as absolute damage-accumulated. If so, the orchestrator should divide by the Anansi 1500-charge threshold (DESIGN §6.1) before passing to `update()`.
3. **Cast-start signal on opponent**: `castStartedAtTick` is an optional field for the bot's reactive-dodge logic. Is the hit-confirm pipeline (T-100) already emitting cast-start events the orchestrator can feed into the snapshot, or should T-107 wire this after the fact?
4. **Mirror Thread clone as a second bot**: ADR-0010 says the clone reuses bot infrastructure. Spawning `createAnansiBot({difficulty: 'easy', ...})` at the clone's position is the suggested handoff, but the clone is supposed to ONLY fire basic attacks (no ability, no ultimate). Confirm: should we expose a `createMirrorThreadBot()` with those branches stripped, or have the orchestrator construct a snapshot where `abilityCdMs = ∞` and `ultCharge = 0` to force the same outcome?
5. **MatchScene instantiation**: the brief's integration note says MatchScene.tsx will instantiate both a human InputManager AND a bot at playerIndex 1. Confirmation that the bot's `update(snapshot)` is called inside the sim tick (60 Hz) rather than the render frame (rAF at 120/144 Hz) — otherwise determinism (ADR-0006) drifts from the human pipeline.
