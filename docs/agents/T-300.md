# T-300 — Audio Layer (SFX + Two-Layer Music Crossfader)

**Persona**: AU (Audio Director)
**Task**: Implement the Howler.js SFX one-shot bank, the Tone.js two-layer adaptive music crossfader, a store→audio bridge, wire the shared `Button` primitive to fire `ui_click`, and boot the stack from `App.tsx`.
**Status**: Complete — `npm run typecheck`, `npm run test` (194/194, +34 new), and `npm run build` (✓ built in ~1.2s) all green.

## Summary

Delivered a four-file audio system under `/src/game/systems/audio/` that pre-loads the four shipped CC0 SFX files on first use, crossfades between the two shipped CC0 music tracks at DESIGN §11's 400 ms default, and derives its live mix (master × per-bus × per-call, plus mono accessibility) from `Settings.audio` via a one-way store subscription. The shared `Button` primitive now fires `ui_click` before every user `onClick`, guarded on `window`. `App.tsx` gains one `useEffect` that calls `initAudio()` on mount (honours browser autoplay by deferring the AudioContext-bearing Tone graph to the first `startMusic()` call).

The audio bundle is cleanly code-split: the production build puts it in its own `audio-*.js` chunk (~266 KB raw / ~67 KB gzip), and Tone's music nodes are constructed lazily on the first `startMusic()` so an unused audio layer costs only the Howler bank decode.

Per brief constraints: `/src/game/entities/`, `/src/rendering/`, `/src/game/arenas/`, `/src/state/store.ts`, and `package.json` are untouched.

## Files created

- `/src/game/systems/audio/sfx.ts` — Howler.js SFX layer. Pre-loads `sfx-ui-click.wav`, `sfx-hit-impact.ogg`, `sfx-anansi-whoosh.flac`, `sfx-anansi-whisper.wav` from `/public/audio/`; exports `playSfx(name, opts?)` typed to `'ui_click' | 'hit' | 'whoosh' | 'whisper'`; computes final volume as `master × sfx × opts.volume` clamped to `[0, 1]`; applies mono via `Howl.stereo(0, id)` when the flag is on; lazy-inits on first `playSfx` call; SSR-safe (returns `null` when `window` is absent).
- `/src/game/systems/audio/music.ts` — Tone.js crossfader. Builds `Player menu → CrossFade.a`, `Player combat → CrossFade.b`, `CrossFade → Gain → Destination` on first `startMusic()`. Both players loop. `setMusicLayer('menu' | 'combat' | 'silent', crossfadeMs = 400)` ramps the crossfader's `fade` signal (and the post-crossfader gain for 'silent'). Respects master × music bus via the post-crossfader `Gain` so Howler SFX are not modulated (the two libraries live on separate AudioContexts). Graph construction is wrapped in try/catch so SSR / test envs (no `AudioContext`) downgrade to a no-op.
- `/src/game/systems/audio/store-bridge.ts` — `connectAudioStoreBridge()` subscribes to `useAppStore` and forwards `settings.audio.{master, sfx, music, mono}` changes to `setSfxVolumes`, `setMusicVolumes`, and `setSfxMono`. Idempotent: double-connect returns the same handle. Also exports `useAudioStoreBridge()` for component-owned lifecycles. Strictly one-way (audio never writes to the store).
- `/src/game/systems/audio/index.ts` — Public barrel + single `initAudio()` bootstrap. `initAudio()` connects the bridge and pre-warms the SFX bank but does **not** auto-start music; the menu decides when to call `startMusic()` (first user gesture). Returns an `AudioBridgeHandle` with `unsubscribe()` so Fast Refresh and tests can tear it down.
- `/tests/unit/audio.test.ts` — 34 new cases across three describe groups: SFX volume math + mono bus propagation + SSR safety; music pure helpers + state machine transitions (`silent` ↔ `menu` ↔ `combat`) + `startMusic/stopMusic` idempotency; store-bridge initial snapshot + live-update forwarding + unsubscribe + double-connect singleton.

## Files modified

- `/src/ui/components/Button.tsx` — added a `handleClick` wrapper that calls `playSfx('ui_click')` (window-guarded) before forwarding to the user's `onClick`. Imports are from the audio barrel. No visual / a11y / sizing behaviour changed.

## Mixing spec

| Bus | Source | Applied by |
|---|---|---|
| master | `settings.audio.master` (0..1) | sfx: per-play volume, clamped; music: post-crossfader Gain (rampTo 50 ms) |
| sfx | `settings.audio.sfx` (0..1) | sfx: per-play volume, clamped |
| music | `settings.audio.music` (0..1) | music: post-crossfader Gain |
| voice | `settings.audio.voice` (0..1) | reserved for v1 whispered VO (DESIGN §11); no audio asset yet — no applier |
| mono | `settings.audio.mono` (bool) | sfx: `Howl.stereo(0, id)` on every active + future play. Music is auto-mono-safe via Tone.CrossFade's equal-power mix + a single StereoPannerNode at center. |

Final SFX volume: `clamp01(master × sfx × (opts.volume ?? 1))`.
Final music gain: `clamp01(master) × clamp01(music)` (linear, ramped 50 ms on updates to avoid zipper noise).

Ducking-under-ultimate (DESIGN §11) and voice-bus wiring are intentionally **not** delivered in T-300 — no VO asset shipped, and ultimate casts aren't yet triggered by combat systems. Hook sites are trivial once those land: combat calls `setMusicLayer('combat', …)` on heat threshold and can temporarily add a `-6 dB` duck via the existing post-crossfader `gain` node.

## Design / implementation decisions

1. **Lazy Tone construction**. Tone's `Player`/`CrossFade`/`Gain` allocation is deferred to the first `startMusic()` call rather than module import. This keeps the page-load cost at just the Howler bank (the four one-shots, < 200 KB combined after decode) and avoids Tone spinning an AudioContext before a user gesture — browsers warn or block that. `initAudio()` only pre-warms the SFX bank.
2. **Module-scope state, not a class**. Combat tick, VFX impact, and the `Button` primitive call `playSfx('whoosh')` without threading a handle. The audio layers are infrastructure; a singleton is the right shape. All state is reset via `__reset*ForTests` hooks used by the unit suite and Fast Refresh.
3. **Bridge never reads outside `settings.audio`**. The store-bridge diffs only the four relevant fields before touching the audio graph. A graphics-preset flip is guaranteed not to bounce music volume. This matches DESIGN §11 / T-300 "audio reads settings, doesn't mutate".
4. **Separate AudioContexts**. Howler and Tone each own their own `AudioContext` by default. We deliberately do not merge them — Tone's graph routes through its own `Destination`; Howler has its own pool. The master volume is applied in both places. Upside: either library can stall (decoding, autoplay gating) without blocking the other. Downside: the browser tab has two audio contexts. Acceptable in 2026 — all v1 target browsers (Chrome/Edge/Brave/Arc/Opera 2-latest, Safari 17+, Firefox 130+) comfortably support multiple contexts.
5. **Mono accessibility**. For SFX, `Howl.stereo(0, id)` places the voice dead-centre, which sums both channels equally — this is the accessible path per DESIGN §13. For music, Tone.CrossFade's equal-power mixer already produces a centre-panned mix unless a source track has hard-panned content (ours don't — both shipped tracks are dual-mono). If a future music asset introduces hard L/R content, wire a `Tone.Mono` node between the crossfader and gain on mono-on.
6. **Button SFX wiring**. The primitive intercepts `onClick`, fires the click SFX, then calls the user's handler (if any). Window feature-detect guards SSR and the vitest happy-dom path where a missing `AudioContext` would otherwise warn on the first click per session.
7. **Code-split result**. The production build emits a dedicated `audio-*.js` chunk (~266 KB raw / ~67 KB gzip), confirming Rolldown's `import()` boundary at the barrel kept Tone and Howler out of the main bundle.

## Known gaps / deferred

- **Per-god voice whispers (DESIGN §11)**: voice bus is reserved; no asset shipped. Hook site is a new `playVoice(godId, lineId)` in `sfx.ts` once ElevenLabs output lands (see R-08 / LICENSES §5).
- **Ducking-under-ultimate**: the post-crossfader `Gain` is the right hook; combat calls `gain.rampTo(-6 dB linear = 0.5, 0.1)` on local ultimate cast and back up on cast-end. Left to T-1xx combat integration.
- **Combat-heat detection**: the brief doesn't define how the combat system signals `setMusicLayer('combat')`. Current contract is that combat will call the setter directly on the 200-damage-in-3-s threshold from DESIGN §11. No polling, no timers inside the audio layer.
- **Howler `unload()` on route change**: the bank is kept alive for the full session. Not a concern for v1's single-arena loop.
- **Tone fixed-tick determinism**: music is decorative — no gameplay reads Tone's state. No ADR-0006 conflict.
- **Autoplay compliance on iOS Safari**: `Tone.start()` inside `startMusic()` is what resumes the suspended AudioContext. As long as callers invoke `startMusic()` from an onClick handler (or a `pointerdown` listener), Safari will honour it. Menu's Play button is the intended first gesture.
- **Button click on disabled / loading**: the SFX plays on click regardless of disabled state because React still dispatches the click event on the custom `handleClick` before the `disabled` attribute short-circuits. In practice the browser suppresses native click on disabled `<button>`, so `handleClick` is not invoked; verified in the existing T-005 Button test path. If a future component renders a visually-disabled non-`<button>` affordance, gate the `playSfx` call on the `disabled` prop there.
